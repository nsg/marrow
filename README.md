## About

Marrow is a model-agnostic agent framework that dynamically generates the tools it needs, self-heals when things break, and keeps everything observable. No hardcoded integrations. No black boxes.

It understands intent through conversation, assembles lean per-task context, and acts — generating sandboxed Lua connectors on the fly for any service it needs to reach. Connect it to Discord, run it from the terminal, or build your own frontend on the library.

## Features

- **Zero hardcoded integrations** — all connectors are Lua scripts generated by the model at runtime, sandboxed and validated
- **Smart triage** — a fast model call decides if external data is needed before running the tool pipeline, skipping unnecessary work for conversational follow-ups
- **Self-healing janitor** — background process reviews generated tools, catches issues (hardcoded values, missing error handling, description mismatches), regenerates fixes, and escalates when it can't repair
- **Tool composition** — tools call other tools via `run_tool()` in Lua, keeping composition logic in testable code with recursion depth capped at 5
- **Working memory** — learns facts from interactions, recalls relevant context per task, avoids duplicates
- **Conversation history** — interactive sessions maintain message history with automatic summarization
- **Model routing** — role-based config maps tasks to models (fast, code, default), all via TOML
- **Full transparency** — structured JSONL event log with progressive detail (`--verbose` for full stream)
- **Multiple frontends** — the core is a library; the CLI and Discord bot are thin wrappers

## Architecture

Marrow is a Cargo workspace with three crates:

```
marrow/            Pure library — agent loop, sandbox, toolbox, memory, routing
marrow-cli/        CLI binary (installed as `marrow`) — REPL and single-prompt mode
marrow-discord/    Discord gateway bridge — responds to mentions and DMs
```

The agent loop:

```
User Input
  → Triage: needs external data? (yes/no)
  → If yes: Tool Selection + Parameter Extraction
    → If no match: Code Generation → Test Run → Save to Toolbox
  → Memory Retrieval (relevant facts)
  → Context Assembly (Lua sandbox + memories)
  → Model Router → Backend (Ollama / OpenAI-compatible)
  → Response
  → Memory Writer (auto-save new facts)
  → Janitor (async background validation)
```

| Component | Description |
|---|---|
| **Triage** | Fast model decides if tools are needed before selection |
| **Tool Selection** | Model picks tools + extracts parameters from task description |
| **Code Generation** | Model writes Lua scripts when no existing tool matches |
| **Lua Sandbox** | Secure runtime — all I/O through controlled Rust host functions (`http_get`, `http_post`, `json_parse`, `json_encode`, `log`, `run_tool`) |
| **Toolbox** | Directory of validated Lua scripts with TOML metadata |
| **Janitor** | Async reviewer — validates code matches description, regenerates on failure, escalates after 3 attempts |
| **Working Memory** | JSON files (one fact per file), model-selected per task, auto-saved post-interaction |
| **Session History** | Per-session message history with automatic summarization at threshold |
| **Event Log** | JSONL append-only log of all pipeline events |
| **Model Router** | Role-based dispatch (fast/code/default) to Ollama and OpenAI-compatible backends |

## Quick Start

```sh
# Build all crates
cargo build --workspace

# Copy and edit config
cp config.example.toml config.toml
# Set your model, provider, and API key in config.toml

# Interactive mode
cargo run -p marrow-cli

# Single prompt
cargo run -p marrow-cli -- -p "What is the weather in Tokyo?"

# Verbose mode (full event stream)
cargo run -p marrow-cli -- --verbose -p "What time is it?"
```

## Discord Bot

The Discord bridge connects Marrow to Discord as a bot that responds to @mentions and DMs.

```sh
cargo run -p marrow-discord
```

Configure the bot in `config.toml`:

```toml
[discord]
token = "your-bot-token"
# toolbox = "toolbox"
# memory = "memory"
# log = "events.jsonl"
# verbose = false
```

The bot needs the **Message Content** privileged intent enabled in the [Discord Developer Portal](https://discord.com/developers/applications).

All settings can be overridden with environment variables (`DISCORD_TOKEN`, `MARROW_TOOLBOX`, `MARROW_MEMORY`, `MARROW_LOG`, `MARROW_VERBOSE`). Env vars take precedence over config file values.

## Configuration

Model routing is configured in `config.toml`. Three roles control which model handles each part of the pipeline:

```toml
[roles.default]    # Chat and task execution (smartest model)
provider = "ollama"
model = "glm-5:cloud"
api_base = "https://ollama.com"
api_key = "your-key"

[roles.fast]       # Triage, tool selection, memory (fast but capable)
provider = "ollama"
model = "kimi-k2.5:cloud"
api_base = "https://ollama.com"
api_key = "your-key"

[roles.code]       # Lua generation and janitor reviews
provider = "ollama"
model = "glm-5:cloud"
api_base = "https://ollama.com"
api_key = "your-key"
```

Supported providers:
- **`ollama`** — Ollama Cloud or local. Omit `api_base` for local at `localhost:11434`. Omit `api_key` for local instances.
- **`openai`** — Any OpenAI-compatible API (OpenAI, OpenRouter, Together, Groq, etc.). Requires `api_key`.

## CLI Reference

| Flag | Default | Description |
|---|---|---|
| `-c, --config` | `config.toml` | Path to config file |
| `-r, --role` | `default` | Model role for task execution |
| `-p, --prompt` | — | Single prompt (non-interactive) |
| `-t, --toolbox` | `toolbox` | Path to toolbox directory |
| `-m, --memory` | `memory` | Path to memory directory |
| `-v, --verbose` | off | Show full event stream |
| `--log` | `events.jsonl` | Path to event log file |
